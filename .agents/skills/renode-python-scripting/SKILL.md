# Skill: Implementing Renode Peripherals using Python Scripts

## Overview

Renode provides a mechanism to rapidly prototype and mock memory-mapped peripherals using Python (specifically, IronPython embedded within Renode's C# environment). This is highly effective for testing firmware against components like custom sensors, motor drivers, or proprietary ICs before the actual Verilog/VHDL or C# peripheral models are fully developed.

This guide covers how to bind a Python script to a memory address space and handle read/write transactions generated by the CPU.

## 1. Platform Setup (`.repl` mapping)

To attach a Python script to the system bus, you must define it in your `.repl` (Renode Platform) file. You will use the `Python.PythonPeripheral` class.

```repl
// Example: Mocking a custom sensor or control register block at 0x40000000
custom_peripheral: Python.PythonPeripheral @ sysbus 0x40000000
    size: 0x100
    initable: true
    filename: "scripts/peripherals/custom_peripheral.py"
```

* **`size`**: Defines the memory footprint reserved on the bus.
* **`initable`**: Ensures the script's initialization logic is run when the machine resets.
* **`filename`**: The relative or absolute path to your Python script.

## 2. Python Script Anatomy

When the CPU accesses the memory region mapped to your peripheral, Renode halts execution and passes a `request` object to your Python script. The script evaluates the request and dictates the response.

### The `request` Object
The `request` object exposes several key properties:
* `request.IsRead` (Boolean): `True` if the CPU is reading from the peripheral.
* `request.IsWrite` (Boolean): `True` if the CPU is writing to the peripheral.
* `request.IsInit` (Boolean): `True` if the peripheral is being initialized (e.g., during machine reset).
* `request.Offset` (Integer): The memory offset being accessed, relative to the base address (e.g., `0x0`, `0x4`).
* `request.Value` (Integer):
    * *On Write*: Contains the data the CPU is writing.
    * *On Read*: Your script must assign a value to this property to return it to the CPU.

### Basic Structure

```python
# custom_peripheral.py

# Register Offsets
REG_ID       = 0x00
REG_CONTROL  = 0x04
REG_STATUS   = 0x08

# Peripheral State
if request.IsInit:
    # Initialize state variables when the machine resets
    state = {
        "control": 0x00000000,
        "status": 0x00000001 # e.g., Bit 0 indicates 'Ready'
    }

elif request.IsRead:
    if request.Offset == REG_ID:
        request.Value = 0xDEADC0DE # Return a mock device ID
        
    elif request.Offset == REG_CONTROL:
        request.Value = state["control"]
        
    elif request.Offset == REG_STATUS:
        request.Value = state["status"]
        
    else:
        # Unmapped read
        self.Log(LogLevel.Warning, "Read from unhandled offset: 0x%X" % request.Offset)
        request.Value = 0x0

elif request.IsWrite:
    if request.Offset == REG_CONTROL:
        state["control"] = request.Value
        self.Log(LogLevel.Info, "Control register updated to: 0x%X" % request.Value)
        
        # Trigger internal behavior based on control bits
        if request.Value & 0x1: 
            state["status"] |= 0x2 # Set a 'Busy' bit in the status register
            
    else:
        # Unmapped or read-only write
        self.Log(LogLevel.Warning, "Write to unhandled/RO offset: 0x%X with value: 0x%X" % (request.Offset, request.Value))
```

## 3. Advanced Behaviors

### Persisting State
Because the Python script is executed dynamically on every bus access, local variables will not persist between transactions. You must use global variables or attach a dictionary to the global scope (like the `state` dictionary in the example above) during the `request.IsInit` phase to persist data between reads and writes.

### Logging and Debugging
The Python script operates within the context of the peripheral object, which exposes Renode's C# logging framework via the `self` keyword.

```python
self.Log(LogLevel.Error, "An error occurred during a read operation.")
self.Log(LogLevel.Info, "Register initialized.")
```
Ensure you have the appropriate `LogLevel` imported or use integer equivalents if the enum is not strictly in scope (though Renode's IronPython environment usually bridges these enums automatically).

### Triggering Interrupts
While pure Python peripherals are generally passive (responding to bus access), you can interact with the machine's IRQ subsystem if you need to signal the CPU. This requires mapping an IRQ in your `.repl` and accessing it via the `self` context, though this borders on the limits of standard Python peripherals and is often better handled by transitioning to a compiled C# model.

## 4. Limitations and Best Practices

1.  **Performance:** Python peripherals incur a significant performance penalty due to the context switch between the emulator core and the IronPython engine. They are best used for slow, infrequent accesses (like configuration registers) or initial prototyping. For high-frequency interrupts or data streaming (e.g., DMA, high-baud UARTs), you should port the Python logic to a native C# Renode peripheral.
2.  **Bitwise Operations:** Ensure you handle the size of the reads/writes correctly. The `request` object typically represents 32-bit (double word) accesses by default on 32-bit architectures, but depending on the bus, you may need to handle byte or half-word masking explicitly in your script if the firmware uses `LDRB` or `LDRH` instructions.
3.  **Strict Typing:** Remember that IronPython bridges Python to C#. When assigning `request.Value`, ensure it is cast or treated as a proper integer to avoid runtime cast exceptions inside the emulator loop.

## 5. System Bus Hooks and Introspection

Beyond standalone peripherals, you can use Python to intercept transactions on existing peripherals (like Timers or UARTs) using **System Bus Hooks**. This is the primary method for implementing cross-peripheral synchronization (e.g., Timer Master/Slave).

### API Discovery (Introspection)
If you are unsure of the available methods or peripheral names, use a script to introspect the `self` (Monitor) and `machine` (Core) objects:

```python
# explorer.py
print("Machine members: %s" % dir(self.Machine))
print("All Names: %s" % list(self.Machine.GetAllNames()))
```

### Hook Attachment
To avoid common IronPython issues with C# generic methods (like `TryGetByName<T>`), use the indexing operator and the Monitor's command parser:

```python
# timer_sync.py
# 1. Get the peripheral object
timer = machine["sysbus.timers1"]

# 2. Attach a hook using monitor.Parse (more robust than direct method calls)
# This hook runs BEFORE any write to any register in the timer
hook_code = "print('Writing to timer offset ' + str(offset))"
monitor.Parse("sysbus SetHookBeforePeripheralWrite timers1 \"%s\"" % hook_code)
```

### Scheduling Virtual Time Actions
To implement time-delayed behaviors (like a TRGO event falling after N ticks), use the machine's scheduler:

```python
from Antmicro.Renode.Time import TimeInterval

def fire_trigger():
    # Peripheral logic here...
    pass

# Schedule in 500ns of virtual time
machine.ScheduleAction(TimeInterval.FromNanoseconds(500), fire_trigger)
```
